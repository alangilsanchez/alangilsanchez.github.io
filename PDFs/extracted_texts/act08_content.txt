Introducción  
SQL Injection es una de las vulnerabilidades más relevantes en el ámbito de la seguridad web, ya 
que permite alterar consultas hacia la base de datos mediante la manipulación de entradas no 
validadas correctamente. A través de este tipo de ataque, un atacante puede consultar información 
sensible, omitir mecanismos de autenticación, modificar registros e incluso comprometer por 
completo la confiden cialidad e integridad de la aplicación. Por esta razón, SQL Injection continúa 
siendo un tema fundamental dentro de las pruebas de penetración y del desarrollo seguro de 
software.  
El presente documento técnico -profesional tiene como finalidad documentar de  manera 
estructurada la resolución de los laboratorios oficiales de SQL Injection de la plataforma PortSwigger 
Web Security Academy, empleando Burp Suite como herramienta principal para la interceptación, 
modificación y análisis de peticiones HTTP. A lo la rgo del desarrollo se presenta el fundamento 
teórico del ataque, la clasificación de sus variantes, el procedimiento práctico seguido en cada 
laboratorio, la explicación de los payloads utilizados, así como las medidas de mitigación 
recomendadas para preve nir este tipo de vulnerabilidades en aplicaciones web.  
Asimismo, este trabajo busca demostrar no solo la ejecución práctica de los laboratorios, sino 
también la comprensión técnica de la lógica detrás de cada explotación, el impacto de seguridad 
que repres enta una inyección SQL y la importancia de aplicar controles defensivos como consultas 
parametrizadas, validación de entradas y principios de mínimo privilegio. De esta manera, el 
documento integra tanto el enfoque ofensivo de identificación y explotación,  como la visión 
defensiva orientada a la protección de sistemas y datos.

Funcionamiento de bases de datos relacionales  
Las bases de datos relacionales son sistemas diseñados para almacenar, organizar y consultar 
información de manera estructu rada mediante tablas. Cada tabla se compone de filas y columnas: 
las filas representan registros individuales, mientras que las columnas representan los atributos o 
campos de dichos registros. Por ejemplo, en una aplicación web puede existir una tabla de u suarios 
con columnas como id, username, password y email, donde cada fila corresponde a un usuario 
distinto.  
El modelo relacional permite que diferentes tablas se vinculen entre sí mediante claves primarias y 
claves foráneas. Una clave primaria identifica de forma única cada registro dentro de una tabla, 
mientras que una clave foránea establece relaciones con registros de otra tabla. Gracias a esto, las 
aplicaciones pueden consultar datos relacionados sin necesidad de duplicar información. Por 
ejemplo, una tienda en línea puede tener una tabla de productos y otra de categorías, 
relacionándolas para mostrar los artículos de una categoría específica.  
Para interactuar con una base de datos relacional se utiliza el lenguaje SQL (Structured Query 
Language), el cu al permite realizar operaciones como consultar, insertar, actualizar y eliminar datos. 
Cuando una aplicación web recibe información del usuario, como credenciales de inicio de sesión o 
filtros de búsqueda, normalmente construye consultas SQL para recuperar  la información 
correspondiente desde la base de datos. Si esta interacción no se implementa de forma segura, un

atacante puede manipular la consulta original e introducir instrucciones maliciosas, dando lugar a 
vulnerabilidades como SQL Injection.  
La impo rtancia de comprender el funcionamiento de las bases de datos relacionales dentro de la 
ciberseguridad radica en que la mayoría de las aplicaciones web modernas dependen de este 
modelo para gestionar información sensible. Por ello, conocer cómo se estructu ran las tablas, cómo 
se relacionan y cómo se ejecutan las consultas SQL es fundamental para entender tanto el origen 
como la explotación de vulnerabilidades en el acceso a los datos.

¿Qué es SQL Injection?  
SQL Injection es una vulnerabilidad de seguridad que ocurre cuando una aplicación web inserta 
directamente datos proporcionado s por el usuario dentro de una consulta SQL sin aplicar 
mecanismos adecuados de validación o sanitización. Como consecuencia, un atacante puede 
modificar la estructura de la consulta original y ejecutar instrucciones no previstas por el 
desarrollador. Esto  permite alterar la lógica de autenticación, acceder a información confidencial, 
enumerar tablas y columnas, o incluso comprometer por completo la base de datos.  
Este tipo de ataque aprovecha la confianza indebida que la aplicación deposita en la entrada d el 
usuario. Por ejemplo, si un sistema concatena directamente un valor ingresado en un formulario de 
inicio de sesión o en un filtro de categoría, el atacante puede introducir caracteres especiales, 
operadores lógicos o sentencias SQL adicionales. De esta manera, la base de datos interpreta la 
entrada maliciosa como parte legítima de la consulta, en lugar de tratarla únicamente como un dato.  
La gravedad de SQL Injection radica en que puede afectar directamente los tres pilares de la 
seguridad de la informac ión. En términos de confidencialidad, permite acceder a datos sensibles 
como usuarios, contraseñas o información personal. En cuanto a la integridad, puede facilitar la 
modificación o eliminación de registros. Respecto a la disponibilidad, un atacante pued e provocar 
errores, saturación del sistema o incluso dañar la operación de la base de datos. Por ello, SQL 
Injection ha sido históricamente considerada una de las vulnerabilidades más críticas en entornos 
web.  
Dentro de las pruebas de penetración, SQL Injection es una técnica fundamental porque demuestra 
cómo una falla aparentemente simple en la validación de entradas puede derivar en un compromiso 
total del sistema. Además, sigue siendo una vulnerabilidad relevante en marcos de referen cia 
reconocidos como OWASP Top 10, debido a su impacto y frecuencia en aplicaciones inseguras. Por 
esta razón, comprender su funcionamiento no solo permite explotarla en un entorno controlado de 
laboratorio, sino también diseñar mejores estrategias de prev ención y endurecimiento de 
aplicaciones reales.  
Clasificación de ataques SQLi  
Los ataques de SQL Injection pueden clasificarse de acuerdo con la forma en que el atacante 
introduce la consulta maliciosa y obtiene la información del servidor. Esta clasificac ión permite 
comprender mejor el comportamiento de cada técnica, su nivel de complejidad y el tipo de 
evidencia que puede observarse durante una prueba de seguridad. En términos generales, las 
variantes más conocidas son In -band, Blind y Out -of-band.  
In-band (Union -based, Error -based)  
La inyección SQL de tipo In -band ocurre cuando el atacante utiliza el mismo canal de comunicación 
para enviar la carga maliciosa y recibir la respuesta del servidor. Es decir, la explotación y la 
extracción de datos se realizan  directamente a través de la respuesta HTTP de la aplicación. Esta es 
una de las formas más comunes y fáciles de identificar dentro de laboratorios y entornos 
vulnerables, ya que los resultados suelen mostrarse de forma inmediata en la página.  
Dentro de es ta categoría se encuentran dos subtipos principales:  
 
--- PAGE 7 ---
→ Union -based SQL Injection . 
Se basa en el uso del operador UNION para combinar la consulta original de la aplicación 
con una consulta inyectada por el atacante. Su propósito es mostrar en la misma respuesta 
datos de otras tablas, como nombres de usuario, contraseñas o versiones del motor de base 
de datos. Para que funcione correctamente, ambas consultas deben devolver el mismo 
número de columnas y tipos de datos compatibles.  
 
→ Error -based SQL Injecti on. 
En esta técnica, el atacante provoca errores intencionales en la base de datos para que el 
sistema revele información útil en los mensajes de error. Si la aplicación muestra detalles 
internos del motor SQL, dichos errores pueden exponer nombres de tabl as, columnas, 
estructura de la consulta o incluso datos sensibles. Esta técnica depende de una 
configuración insegura que permita mostrar errores detallados al usuario.  
Blind (Boolean -based, Time -based)  
La inyección SQL de tipo Blind  se presenta cuando la aplicación no devuelve mensajes de error 
visibles ni muestra directamente los resultados de la consulta maliciosa. En este escenario, el 
atacante debe inferir la información observando cambios indirectos en la respuesta del sistema, 
como variaciones en el contenido, el comportamiento o el tiempo de respuesta. Aunque este tipo 
de explotación suele ser más lenta, sigue siendo altamente peligrosa.  
Sus dos variantes más comunes son:  
→ Boolean -based Blind SQL Injection.  
Consiste en enviar condiciones lógicas que p rovocan respuestas diferentes según si la 
expresión es verdadera o falsa. Por ejemplo, la página puede mostrar un mensaje como 
“Welcome back” cuando la condición se cumple, y omitirlo cuando no se cumple. A partir 
de estas diferencias sutiles, el atacante puede extraer información carácter por carácter.  
 
→ Time -based Blind SQL Injection.  
En esta variante, el atacante introduce funciones que obligan a la base de datos a retrasar 
su respuesta durante un número específico de segundos. Si la respuesta tarda exact amente 
el tiempo esperado, se confirma que la condición inyectada fue verdadera. De esta forma, 
el tiempo de respuesta se convierte en el canal para inferir datos aun cuando la página no 
muestre resultados visibles.  
 
Out-of-band  
La inyección SQL Out-of-band ocurre cuando la extracción de información no se realiza a través de 
la misma respuesta HTTP de la aplicación, sino por un canal externo controlado por el atacante, 
como DNS o HTTP. Este tipo de ataque suele utilizarse cuando no es posible obtener datos mediante 
respuestas directas ni mediante diferencias visibles o temporales.  
Aunque es menos común que las variantes anteriores, puede ser muy efectiva en escenarios donde 
el servidor tiene conectividad hacia servicios externos. Por ejemplo, una carga malic iosa puede 
forzar a la base de datos a realizar una consulta DNS hacia un dominio controlado por el atacante,

enviando en esa petición información sensible codificada. Debido a ello, este tipo de SQL Injection 
representa un riesgo importante en infraestruc turas mal segmentadas o con configuraciones 
permisivas.  
Comprender la clasificación de los ataques SQL Injection es fundamental para interpretar 
correctamente el comportamiento de cada laboratorio, seleccionar la técnica de explotación 
adecuada y proponer medidas defensivas acordes al tipo de vulnerabilidad identificada. Cada 
variante demuestra que una mala validación de entradas puede convertirse en una puerta de acceso 
crítica hacia la base de datos y, en consecuencia, hacia la información sensib le de una aplicación 
web.  
Impacto en la confidencialidad, integridad y disponibilidad  
La vulnerabilidad de SQL Injection puede afectar directamente los tres principios fundamentales de 
la seguridad de la información: confidencialidad, integridad y disponib ilidad. Esto se debe a que la 
explotación de una inyección SQL permite interactuar de forma no autorizada con la base de datos, 
que normalmente contiene información crítica para el funcionamiento de la aplicación.  
En cuanto a la confidencialidad, el impact o ocurre cuando un atacante logra consultar datos que no 
debería conocer, como credenciales de acceso, información personal de usuarios, correos 
electrónicos, historiales, registros internos o datos administrativos. En varios laboratorios de esta 
práctica fue posible extraer nombres de usuario y contraseñas, lo que demuestra cómo una mala 
validación de entradas puede terminar exponiendo información sensible.  
Respecto a la integridad, SQL Injection también puede permitir la alteración de la información 
almac enada. Si la cuenta de base de datos utilizada por la aplicación tiene privilegios suficientes, un 
atacante podría modificar registros, cambiar contraseñas, insertar datos falsos o eliminar 
información existente. Esto compromete la confiabilidad de los dat os y afecta directamente la 
operación del sistema.  
En términos de disponibilidad, una inyección SQL puede provocar fallos en la aplicación, errores 
internos del servidor, bloqueos de procesos o consultas costosas que consuman recursos excesivos. 
Incluso, e n escenarios más graves, podría utilizarse para borrar tablas o afectar la estructura de la 
base de datos, dejando el servicio fuera de operación. Por ello, SQL Injection no solo representa un 
problema de exposición de información, sino un riesgo integral para todo el sistema.  
Relación con OWASP Top 10  
La vulnerabilidad de SQL Injection se encuentra estrechamente relacionada con los riesgos más 
críticos identificados por OWASP Top 10, una de las referencias más importantes en seguridad de 
aplicaciones web. OWASP, cuyas siglas corresponden a Open Worldwide Application Security 
Project, mantiene esta clasificación con el propósito de concientizar sobre las fallas más comunes y 
peligrosas que afectan a los sistemas modernos. Dentro de este marco, SQL Injection ha sido 
históricamente una de las vulnerabilidades más representativas.  
En versiones anteriores del OWASP Top 10, los ataques de inyección aparecían de forma explícita 
como una categoría principal debido a su alta frecuencia e impacto. Aunque en versiones más 
recientes la clasificación se ha ampliado y reorganizado, SQL Injection continúa formando parte del 
grupo de vulnerabilidades relacionadas con fallas de validación, interpretación insegura de entradas

y ejecución no intencionada de comandos sobre siste mas internos. Esto demuestra que, a pesar de 
los avances en frameworks y buenas prácticas de desarrollo, la inyección sigue siendo una amenaza

La razón por la que SQL Injection ocupa un lugar tan importante dentro de OWASP es que una sola 
vulnerab ilidad de este tipo puede escalar rápidamente a una afectación completa del sistema. A 
diferencia de otras fallas que requieren múltiples condiciones para ser explotadas, una inyección

razón, el análisis de SQL Injection y sus mecanismos de defensa constituye una competencia esencial 
para cualquier estudiante o profesional orientado a la seguridad i nformática, auditoría de

Importancia en pruebas de penetración  
SQL Injection es una de las vulnerabilidades más importantes dentro de las pruebas de penetración, 
ya que permite evaluar de forma directa la seguridad en  la interacción entre la aplicación web y la 
base de datos. Su detección demuestra que existe una validación deficiente de entradas y que el 
sistema podría ser manipulado para ejecutar instrucciones no autorizadas.  
Durante un pentest, identificar una vulne rabilidad de SQL Injection tiene un alto valor porque puede 
conducir rápidamente a resultados críticos, como evasión de autenticación, enumeración de tablas 
y columnas, recuperación de credenciales, acceso a cuentas privilegiadas o extracción de 
informació n sensible. Por esta razón, suele considerarse una de las pruebas más relevantes dentro 
de la auditoría de aplicaciones web.  
Además, la explotación controlada de SQL Injection permite medir el impacto real de la falla y 
generar recomendaciones concretas de  remediación. No se trata únicamente de demostrar que la 
vulnerabilidad existe, sino de analizar hasta dónde puede llegar un atacante si la organización no 
implementa consultas parametrizadas, validación de entradas, controles de privilegios y manejo 
segur o de errores.  
Su importancia en pruebas de penetración también radica en que combina análisis técnico, 
comprensión de bases de datos, manipulación de peticiones HTTP y uso de herramientas 
especializadas como Burp Suite. Por ello, el estudio y documentación  de estos laboratorios fortalece 
habilidades esenciales para la evaluación ofensiva y defensiva de aplicaciones web.

Tipos de payloads  
En SQL Injection, un payload es la cadena de texto o instrucción que el atacante inserta en un 
parámetro vulnerable con  el objetivo de alterar el comportamiento de la consulta SQL original. Los 
payloads varían según el tipo de inyección, la base de datos utilizada y el objetivo del ataque, ya sea 
autenticar sin credenciales, identificar columnas, extraer datos o provocar r etrasos en la respuesta.  
Un tipo común son los payloads de evasión de autenticación, utilizados para modificar la lógica de 
una consulta de inicio de sesión. Su propósito es hacer que la condición del WHERE se evalúe como 
verdadera sin necesidad de conocer  la contraseña legítima. Este tipo de payload demuestra cómo 
una autenticación insegura puede ser anulada con una entrada manipulada.  
También existen los payloads UNION -based, que se emplean para combinar la consulta original de 
la aplicación con una consu lta inyectada por el atacante. Estos payloads permiten recuperar 
información de otras tablas, como usuarios, contraseñas, nombres de columnas o versiones del 
motor de base de datos. Suelen incluir valores NULL para identificar el número de columnas y 
despu és reemplazarlos por texto o datos de interés.  
Otro grupo importante son los payloads error -based, diseñados para provocar errores controlados 
en la base de datos y obtener información a partir de los mensajes devueltos por el servidor. Cuando 
la aplicació n expone errores detallados, estos payloads pueden revelar datos sensibles, estructura 
de tablas o resultados de subconsultas.  
En el caso de Blind SQL Injection, se utilizan payloads booleanos y payloads basados en tiempo. Los 
booleanos introducen condicio nes verdaderas o falsas para observar diferencias en la respuesta de 
la página, mientras que los basados en tiempo emplean funciones de retraso para confirmar si una 
condición fue ejecutada correctamente por la base de datos. Aunque estos métodos suelen se r más 
lentos, son muy efectivos cuando la aplicación no muestra errores ni resultados visibles.  
Finalmente, también pueden existir payloads de enumeración y payloads de exfiltración, orientados 
a descubrir tablas, columnas, tipos de datos o extraer múltipl es valores en una sola columna. En 
todos los casos, el payload representa la parte central del ataque, ya que es el medio por el cual se 
manipula la consulta original. Por ello, entender su estructura y propósito es indispensable tanto 
para la explotación en entornos controlados como para la implementación de defensas efectivas.  
Principios de defensa  
La prevención de SQL Injection requiere aplicar controles de seguridad desde el diseño y desarrollo 
de la aplicación, no únicamente después de detectar la vuln erabilidad. La medida más importante y 
efectiva consiste en utilizar consultas parametrizadas o sentencias preparadas, ya que estas separan 
de forma estricta la estructura del comando SQL y los datos introducidos por el usuario. De esta 
manera, aunque un a tacante intente enviar caracteres especiales, operadores lógicos o sentencias 
maliciosas, la base de datos los interpretará únicamente como datos literales y no como 
instrucciones ejecutables.  
Otro principio fundamental es la validación estricta de entradas. Toda información recibida desde 
formularios, parámetros de URL, encabezados HTTP o cookies debe comprobarse antes de ser 
procesada. Esto implica verificar longitud, formato, tipo de dato y v alores permitidos. Cuando es 
posible, se recomienda aplicar esquemas de lista blanca, aceptando únicamente valores 
--- PAGE 11 ---
previamente definidos por la lógica de la aplicación. Aunque la validación por sí sola no sustituye a 
las consultas parametrizadas, sí contri buye a reducir la superficie de ataque.  
También es recomendable utilizar ORMs o frameworks seguros, ya que muchos de ellos incorporan 
mecanismos internos para manejar consultas parametrizadas y abstraer el acceso directo a la base 
de datos. Sin embargo, su  uso no garantiza seguridad automática si el desarrollador implementa 
consultas dinámicas inseguras o concatena manualmente valores sin control. Por ello, el framework 
debe entenderse como un apoyo y no como un sustituto de las buenas prácticas de desarrol lo 
seguro.  
El principio de mínimo privilegio en la base de datos es otra medida crítica. La cuenta utilizada por 
la aplicación web no debería tener permisos innecesarios sobre todas las tablas, vistas o 
procedimientos. Si un atacante logra explotar una iny ección SQL, el alcance del daño dependerá en 
gran parte de los privilegios asociados a esa cuenta. Limitar permisos de lectura, escritura, 
modificación y administración reduce significativamente el impacto potencial de una explotación.  
Finalmente, es impor tante complementar estas defensas con manejo seguro de errores, monitoreo 
de actividad sospechosa y pruebas periódicas de seguridad. Los mensajes de error no deben 
exponer detalles internos del motor de base de datos, ya que pueden facilitar ataques de tip o error -
based. Asimismo, el monitoreo de consultas anómalas y la realización de auditorías permiten 
identificar comportamientos sospechosos antes de que deriven en un incidente mayor. En conjunto, 
estas medidas forman una estrategia defensiva integral cont ra una de las vulnerabilidades más 
peligrosas en aplicaciones web.

Nombre del laboratorio:  
SQL injection UNION attack, determining the number of columns returned by the query  
Nivel:  
Practitioner  
Tipo de SQLi:  
Inyección SQL In -band (basada en UNION)  
Objetivo del laboratorio:  
Mapeo de la estructura (Número de columnas): Nuestro objetivo principal en esta fase no es extraer 
datos (todavía), sino realizar una labor de reconocimiento. Necesitamos determinar el número 
exacto de columnas que devuelve la consulta SQL original de la página. El operador UNION tiene 
una regla estricta: para poder fusionar dos consultas, ambas deben tener exactamente la misma 
cantidad de columnas. Descubrir este número es el requisito indispensable para poder, en pasos 
posteriores, extraer información sensible como tablas de usuarios y contraseñas.  
Explicación técnica de la vulnerabilidad:  
Falta de sanitización y concatenación directa: La falla de seguridad ocurre en cómo el servidor 
procesa el parámetro category de la URL. El backend de la aplicación está tomando el texto que 
seleccionamos en el filtro (por ejemplo, "Food & Drink") y lo está "pegando" o concatenando 
directamente dentro del código de una consulta SQL (probablemente algo parecido a SELECT * 
FROM products WHERE category = 'texto_del_usuario'). Al no limpiar (sanitizar) nuestra entrada, el 
motor de la base de datos es incapaz de distinguir dónde termina el dato legítimo y dó nde empiezan 
nuestras instrucciones maliciosas, ejecutando ambas por igual.

La vulnerabilidad se encuentra en la cookie TrackingId, la cual es utilizada por la aplicación web 
dentro de una consulta SQL s in validación ni sanitización adecuada.  
Esto permite que un atacante modifique el valor de la cookie para introducir instrucciones SQL 
adicionales. En este laboratorio, la consulta maliciosa se utiliza para recuperar la contraseña del 
usuario administrador  desde la tabla users.  
Debido a que la aplicación no muestra resultados ni errores SQL, se utiliza una técnica Out -of-Band, 
donde la información robada se envía a un servidor externo. En este caso, la contraseña se 
concatena dentro de un subdominio que ser á resuelto por el servidor DNS de Burp Collaborator.  
Cuando la base de datos intenta resolver ese dominio, la información incluida en el subdominio 
queda registrada en Burp Collaborator, permitiendo al atacante visualizar los datos extraídos.

Laboratorio  18 
Nombre del laboratorio:  
SQL injection with filter bypass via xml encoding  
Nivel:  
Practitione  
Tipo de SQLi:  
SQL Injection con bypass de filtros mediante codificación XML  
Objetivo del laboratorio:  
El objetivo del laboratorio es explotar una vulnerabilidad de SQL Injection presente en la 
funcionalidad que consulta el stock de los productos.  
La aplicación utiliza un firewall de aplicaciones web que detecta y bloquea intentos de inyección 
SQL. Por lo tanto, el atacant e debe encontrar una forma de evadir este filtro para ejecutar la 
consulta maliciosa.  
Una vez superado el WAF, se debe utilizar una consulta UNION SELECT para extraer los nombres de 
usuario y contraseñas almacenados en la tabla users. Finalmente, se utiliz a la contraseña del usuario 
administrator para iniciar sesión y completar el laboratorio.  
Explicación técnica de la vulnerabilidad:  
La vulnerabilidad se encuentra  en la funcionalidad Check Stock, que consulta la cantidad de 
unidades disponibles de un produ cto en una tienda específica.  
Cuando el usuario presiona esta opción, la aplicación envía una solicitud HTTP que contiene 
parámetros como productId y storeId. Estos valores son utilizados directamente dentro de una 
consulta SQL  sin una validación adecuada.  
Para mitigar ataques, la aplicación cuenta con un WAF, el cual analiza las solicitudes en busca de 
patrones asociados con ataques como SQL Injection. Si detecta caracteres sospechosos como 
comillas, comentarios SQL o palabras clave como UNION SELECT, la solicitud es bloqueada y el 
servidor devuelve un mensaje indicando que se detectó un ataque.  
Sin embargo, el WAF solo analiza el contenido en texto plano. Al codificar los caracteres del payload 
utilizando entidades XML en formato hexadecimal, e l firewall no reconoce el patrón de ataque y 
permite que la solicitud continúe.

se ejecuta normalmente en la base de datos. Esto permite introducir una co nsulta UNION SELECT 
que combina el resultado original con información de la tabla users, donde se almacenan los 
nombres de usuario y contraseñas.

Conclusión  
Este laboratorio nos permitió demostrar cómo una vulnerabilidad de SQL Injection basada en UNION  
puede ser aprovechada en un entorno Oracle para enumerar la estru ctura de la base de datos y 
extraer credenciales almacenadas en la tabla de usuarios. A través de la identificación de tablas, 
columnas y registros sensibles, fue posible recuperar la contraseña del usuario administrator  e 
iniciar sesión de manera exitosa.  La práctica evidenció que, aunque Oracle maneja estructuras 
distintas a otros motores de base de datos, una implementación insegura sigue permitiendo la 
exposición de información crítica. Por ello, concluimos que este tipo de vulnerabilidad representa 
un riesgo grave para la confidencialidad del sistema y confirma la necesidad de aplicar consultas 
parametrizadas, validación de entradas y controles estrictos de acceso a la base de datos.
